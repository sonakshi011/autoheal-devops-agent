# Troubleshooting & Resolution Guide

This document captures **real-world issues encountered and resolved** during the development of the AutoHeal DevOps Agent's full-stack integration. Use this as a guide to debug and maintain the platform.

---

## 🛡️ 1. Security Gate & Directory Mount Issues

### Issue: "Both Trivy and Bandit scan reports are missing from the reports directory" in Docker
*   **Symptom**: When loading the Security page inside Docker Compose, it reports that both Trivy and Bandit scan reports are missing, even though they exist on the host filesystem under `./reports/`.
*   **Root Cause**: The FastAPI production Docker container is built using a minimal, multi-stage distroless base image that does not copy the `./reports` folder (to keep images small and secure). Additionally, `docker-compose.yml` originally did not configure any volume mounts for `/app/reports`. As a result, the container’s virtual directory `/app/reports` remained empty.
*   **Resolution**: Bind-mount the host reports folder to the container's directory in [docker-compose.yml](../docker-compose.yml):
    ```yaml
        volumes:
          - ./reports:/app/reports
    ```
    This links the filesystems dynamically, making any scan file generated by host scripts instantly accessible to the backend runtime.

## 🤖 2. Dynamic Failure Synchronization Issues

### Issue: AI Analysis page shows old/stale failure logs after the build succeeds
*   **Symptom**: When a pipeline run succeeds on GitHub, the AI Analysis page still shows the old root cause, remediation steps, and code suggestions from a previous failure.
*   **Root Cause**: The FastAPI backend endpoint `/api/v1/ai/latest-diagnosis` was statically reading and serving whatever `ai_diagnosis.json` report was present on the disk. It had no mechanism to synchronize its file-serving logic with the live state of the GitHub branch.
*   **Resolution**: Updated the backend route to dynamically inspect the latest workflow run conclusion from PyGithub before serving the report. If the latest run's conclusion is `success`, the backend suppresses the stale filesystem report and immediately throws a `404 Not Found` with a descriptive message. The React client intercepts this and renders the green **"System Fully Healthy"** operational card.

## 📊 3. Telemetry & Grafana iframe Issues

### Issue: Monitoring iframe refuses to connect (`X-Frame-Options: deny`)
*   **Symptom**: The Monitoring page loads the control panel shell, but the embedded Grafana iframe area is blank and displays "localhost refused to connect" or prints `Refused to display '...' in a frame because it set 'X-Frame-Options' to 'deny'` in the browser console.
*   **Root Cause**: By default, Grafana blocks iframe embedding to protect users from clickjacking security exploits, injecting the `X-Frame-Options: deny` header into all HTTP responses.
*   **Resolution**: Overrode the default security parameters by injecting the `GF_SECURITY_ALLOW_EMBEDDING=true` environment variable into the `grafana` container configuration in [docker-compose.yml](../docker-compose.yml).

### Issue: Embedded iframe displays a login prompt instead of the telemetry dashboard
*   **Symptom**: The iframe connects, but asks for the admin username and password, breaking the integrated control panel experience.
*   **Root Cause**: Grafana is designed to require user authentication by default. Without anonymous read access configured, the dashboard view cannot be exposed inside third-party shells without triggering login redirects.
*   **Resolution**: Enabled secure, read-only anonymous authentication by passing the following overrides inside the Grafana container env inside [docker-compose.yml](../docker-compose.yml):
    ```yaml
          - GF_AUTH_ANONYMOUS_ENABLED=true
          - GF_AUTH_ANONYMOUS_ORG_ROLE=Viewer
    ```

### Issue: Iframe shows "Dashboard not found" or "Invalid UID in annotation request"
*   **Symptom**: The iframe renders Grafana's dark theme inside the control panel, but shows a red banner declaring "Dashboard not found".
*   **Root Cause**: Grafana routes dashboards exclusively by their unique identifier (**UID**) inside the URL (e.g. `/d/<uid>/<slug>`). 
    The pre-configured dashboard JSON file [autoheal-overview.json](../monitoring/grafana/provisioning/dashboards/autoheal-overview.json) declared:
    ```json
    "uid": "autoheal-overview-v1"
    ```
    However, the frontend fell back to requesting the path `/d/autoheal-overview`, resulting in a UID mismatch and database resolution failure.
*   **Resolution**: Aligned the default `embedUrl` path inside [monitoring/page.tsx](../frontend/app/monitoring/page.tsx) to point directly to `autoheal-overview-v1`:
    ```typescript
    const embedUrl = `${grafanaUrl}/d/autoheal-overview-v1/autoheal-platform-overview?orgId=1&refresh=5s&theme=dark&kiosk`;
    ```

## 🔌 4. Port & Socket Binding Conflicts

### Issue: Backend crashes on startup with `OSError: [Errno 10048] address already in use`
*   **Symptom**: While starting the backend locally using `python -m uvicorn app.main:app --reload`, the terminal prints a socket bind crash dump and terminates immediately.
*   **Root Cause**: Docker Compose is already running in the background and holding host port `8000` for the `autoheal-api` container service. Only one active socket binding is allowed per IP/port combination.
*   **Resolution**: Either stop the Docker Compose container first (`docker compose stop app`) before initiating local debugging, or start the local python dev server on a different port:
    ```bash
    python -m uvicorn app.main:app --reload --port 8080
    ```
